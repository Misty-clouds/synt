# Synt — Architecture

See **[`architecture.svg`](./architecture.svg)** for the visual diagram. This document
explains the three things judges should be able to read off it: how Synt talks to
**Splunk**, how the **AI/agent** is integrated, and the **data flow** between services.

## The one-sentence model

> **Splunk is both the agent's senses and (partly) its brain; NestJS is its nervous
> system; Next.js is the window you watch it think through.**

- **Splunk = data + tools + reasoning.** Security events live in Splunk indexes. The
  **MCP Server** is the agent's *hands* (search + response actions). **Splunk hosted
  models** are the agent's *brain* (hypotheses, NL→SPL, narrative, MITRE mapping).
- **NestJS (`apps/api`) = the runtime.** It runs the OODA loop, persists state, streams
  reasoning over SSE, renders the PDF case file, and gates response actions.
- **Next.js (`apps/web`) = the Reasoning Theatre.** It renders the live thought stream,
  the self-assembling investigation graph, and the case file with a human approval gate.
- **MongoDB = mutable app state.** Investigations, traces, case files, approvals.
- **Cloudflare R2 = binary artifacts.** The generated PDF case files only.

## Data flow (numbered)

```
                         ┌──────────────────────────────────────┐
                         │              SPLUNK                   │
                         │  indexes (auth, network, endpoint…)   │
                         │  MCP Server (search + actions)        │  ← hands
                         │  Hosted Models (reasoning)            │  ← brain
                         └───────┬───────────────┬──────────────┘
            (1) inject seed      │               │ (2) Act: run SPL,
                via HEC          │               │     response actions
                                 ▼               ▼
                    ┌────────────────────────────────────────┐
                    │            apps/api (NestJS)            │
                    │  TriggerService ─ creates Investigation │
                    │  AgentRunner (packages/agent OODA loop) │
                    │     Orient/Decide ─► Hosted Models      │
                    │     Act           ─► MCP Server         │
                    │  SseGateway (live event stream)         │
                    │  CaseFileService ─► PDF ─► R2           │
                    │  ResponseService ─► MCP (gated)         │
                    └───────┬───────────────────────┬─────────┘
              (3) persist    │                       │ (3) stream events (SSE)
              trace + state  ▼                       ▼
                    ┌──────────────┐      ┌─────────────────────────┐
                    │   MongoDB    │      │   apps/web (Next.js)    │
                    │ investigations│     │  Reasoning Theatre      │
                    │ traces        │     │  self-assembling graph  │
                    │ caseFiles     │     │  Case file + PDF view   │
                    │ approvals     │◄─────┤  Approval gate ─────────┼─► (5) approve → ResponseService → MCP
                    └──────────────┘  REST │  NL command box ────────┼─► hosted model → re-pivot
                          │                └─────────────────────────┘
                          ▼
                    ┌──────────────┐
                    │ Cloudflare R2│  (4) PDF case files (download links)
                    └──────────────┘
```

1. **Trigger / inject.** A Splunk saved-search notable (or the demo "Inject scenario"
   button) seeds events into the right indexes over **HEC** and produces a `Signal`.
   `TriggerService` creates an `Investigation` and dispatches `AgentRunner`.
2. **OODA loop.** `packages/agent` runs Observe → Orient → Decide → Act → Loop. Every
   query is generated by a **hosted model** (NL→SPL) and executed through the **MCP
   Server**; the model reads results and updates confidence. Entities/edges discovered
   in results drive the live graph.
3. **Persist + stream.** Each `TraceEvent` is appended to MongoDB and pushed to the
   `SseGateway`, which the Theatre tails over SSE (replaying history on connect).
4. **Case file.** On completion, `CaseFileService` composes the case file (hosted model
   writes the summary/MITRE/playbook), renders a **PDF**, and stores it in **R2**.
5. **Response gate.** High-confidence, auto-approvable actions self-execute (ransomware);
   everything else waits for a human Approve/Reject, which calls the **MCP Server**.

## Packages (soul-first build order)

| Package | Responsibility |
|---|---|
| `packages/shared` | TypeScript data contracts shared by api + web (single source of truth). |
| `packages/splunk` | `SplunkClient` (real **MCP** + in-memory mock), `HostedModel` (real + deterministic mock), HEC ingest, NL→SPL. |
| `packages/scenarios` | The four attack scenarios: seed generators, trigger signals, ground truth. |
| `packages/agent` | The framework-agnostic OODA engine. `runInvestigation()` yields `TraceEvent`s. |

The agent depends only on the `SplunkClient` + `HostedModel` interfaces, so it runs
**identically** against real Splunk/MCP/hosted-models or the offline mocks. The mocks
make the demo deterministic and let `pnpm --filter @synt/agent verify` prove all four
scenarios reach ground truth with zero external services.

## Offline-first design

With `USE_MOCK_SPLUNK=true` (default) the entire system runs with **no external
dependencies**: an in-memory Splunk, a deterministic hosted-model stand-in, an in-memory
store, and local-disk PDF storage. Setting the real env vars swaps each layer to live:
`SPLUNK_MCP_URL` → real MCP, `SPLUNK_MODEL_ENDPOINT` → real hosted models, `MONGODB_URI`
→ Mongo, R2 creds → R2. The agent, API and web code paths are unchanged.
